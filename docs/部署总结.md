根据这次 **BiliSync 从拉取代码 → 部署 → HTTPS → PM2 → WebSocket 排错** 的全过程，总结出一套更稳的生产部署流程。

你上传的《GitHub 项目拉取方法》内容我已结合本次实际踩坑整理。

# BiliSync 生产部署标准流程（Ubuntu + Nginx + PM2）

---

# 第一阶段：服务器初始化

## 1. 安装基础环境

确认：

```bash
lsb_release -a
```

本次环境：

```
Ubuntu 22.04.5 LTS
```

安装：

```bash
apt update
apt install -y nginx git curl
```

安装 Node：

推荐使用 nvm：

```bash
nvm install 22
nvm use 22
```

确认：

```bash
node -v
npm -v
```

---

# 第二阶段：拉取项目

## 2. 创建部署目录

```bash
mkdir -p /opt/bilisync
cd /opt/bilisync
```

---

## 3. 拉取 GitHub 项目

首次部署：

```bash
git clone https://github.com/xuhuilun/BiliSync.git /opt/bilisync
```

进入：

```bash
cd /opt/bilisync
```

确认：

```bash
git status
```

正常：

```
working tree clean
```

---

# 第三阶段：安装依赖和构建

## 4. 安装依赖

不要直接：

```bash
npm install
```

生产环境优先：

```bash
npm ci
```

原因：

- 使用 package-lock.json
- 保证版本一致
- 避免依赖漂移

---

## 5. 构建项目

查看脚本：

```bash
cat package.json
```

确认：

```json
"scripts": {
    "build": "..."
}
```

执行：

```bash
npm run build
```

生成：

```
server/dist
web/dist
```

---

# 第四阶段：配置生产环境

## 6. 配置环境变量

创建：

```bash
nano .env
```

例如：

```env
NODE_ENV=production

ALLOWED_ORIGINS=https://bilisync.top
```

重点：

生产环境必须配置：

```
ALLOWED_ORIGINS
```

否则启动报错：

```
SecurityConfigError:
ALLOWED_ORIGINS is empty
```

---

# 第五阶段：先启动后端，不急着配置 Nginx

这是本次最大的经验。

不要：

```
Nginx → SSL → Certbot → 前端
```

应该：

```
Node 后端
 ↓
确认健康
 ↓
Nginx代理
 ↓
HTTPS
```

---

## 7. PM2 启动

例如：

```bash
pm2 start server/dist/index.js \
--name bilisync
```

查看：

```bash
pm2 list
```

必须：

```
bilisync online
```

---

## 8. 验证 Node

不要相信 PM2 online。

必须：

```bash
curl http://127.0.0.1:8787/healthz
```

正常：

```json
{
  "ok": true,
  "data": {
    "status": "healthy"
  }
}
```

同时：

```bash
ss -lntp | grep 8787
```

应该看到：

```
node :8787
```

# 第六阶段：PM2 开机启动

执行：

```bash
pm2 startup
```

然后：

```bash
pm2 save
```

否则服务器重启后：

```
PM2服务存在
但是应用不存在
```

# 第七阶段：Nginx 配置

## 关键经验：

### 不要直接套官方 nginx 示例

本次遇到：

### 错误1

```
unknown directive "http2"
```

原因：

服务器：

```
nginx 1.18
```

不支持：

```nginx
http2 on;
```

修改：

```nginx
listen 443 ssl http2;
```

---

## 错误2

Certbot 自动修改导致：

```
HTTPS 无限重定向
```

原因：

错误：

```
443 server
里面 return 301 https
```

正确：

80：

```
http → https
```

443：

```
正常代理
```

---

# 第八阶段：BiliSync 特殊点（重点）

这是本次最大经验。

BiliSync：

不是：

```
Nginx静态网页
+
Node API
```

而是：

```
Node一体化服务
|
├── Web页面
├── API
└── WebSocket
```

证据：

```
server/dist/web-panel.js
```

引用：

```
../../web/dist
```

所以：

Nginx主要负责：

```
HTTPS
WebSocket转发
反向代理
```

---

## WebSocket要求

浏览器：

```
wss://bilisync.top/
```

必须到：

```
127.0.0.1:8787
```

Nginx必须：

```nginx
location / {

proxy_pass http://127.0.0.1:8787;

proxy_http_version 1.1;

proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection $connection_upgrade;

proxy_set_header Host $host;

proxy_read_timeout 86400s;

}
```

否则：

浏览器：

```
WebSocket connection failed
```

---

# 第九阶段：SSL证书

正确顺序：

先保证：

```bash
nginx -t
```

通过。

然后：

```bash
certbot --nginx -d bilisync.top
```

如果因为旧配置失败：

停止：

```bash
systemctl stop nginx
```

申请：

```bash
certbot certonly --standalone -d bilisync.top
```

确认：

```bash
ls /etc/letsencrypt/live/bilisync.top/
```

存在：

```
fullchain.pem
privkey.pem
```

# 本次主要经验教训

| 问题                  | 教训                                   |
| --------------------- | -------------------------------------- |
| 直接套 Nginx 示例失败 | 先确认服务器 Nginx 版本                |
| Certbot导致循环跳转   | 证书申请前先整理 server 块             |
| PM2 online误判        | 必须 curl healthz                      |
| ALLOWED_ORIGINS缺失   | 生产环境必须配置                       |
| WebSocket失败         | BiliSync 根路径就是 WS，不是普通静态站 |
| 过早修改 Nginx        | 先确认 Node 是否负责前端               |
| 部署顺序错误          | 先跑 Node，再接 Nginx                  |

---

最终推荐架构：

```
Internet
   |
   |
HTTPS 443
   |
 Nginx
   |
   |
127.0.0.1:8787
   |
 BiliSync Node
   |
 PM2
```

这套流程适合以后 BiliSync 每次升级和迁移。

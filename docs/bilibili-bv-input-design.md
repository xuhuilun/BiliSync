# B 站 BV 号智能识别设计

## 1. 现状与目标

现有服务端已经具备以下能力：

- 解析纯 BV/AV 输入。
- 解析带追踪参数的 `www.bilibili.com/video/...` 链接。
- 在服务端展开 `b23.tv` 短链，最多跟随 3 次跳转。

本次真正需要补齐的是：统一并收紧 BV 格式、让 Web 输入框即时反馈识别结果，以及统一无效输入提示。短链继续在用户点击“解析”后由服务端处理，不在 `input` 事件中发起网络请求。

## 2. 方案对比

### 方案一：协议层统一纯函数 + 服务端短链展开（推荐）

**核心思路：** 在 `@bili-syncplay/protocol` 增加统一的 B 站输入解析函数。它同步处理纯 BV 和可信 B 站长链接，严格校验 `BV` 后 10 位字母数字；Web 和服务端共同复用。`b23.tv` 只识别为“需要服务端展开”，实际跳转仍由服务端完成。

**优点：**

- BV 格式和 URL 规范化只有一个来源，符合仓库结构约束。
- Web 即时反馈与服务端最终校验不会产生规则漂移。
- 纯函数易测试，不引入前端跨域请求。
- 对现有短链实现改动小。

**缺点：**

- 需要同时修改协议、服务端和 Web 三处消费代码。
- 若改变现有宽松输入的接受范围，需要补回归测试确认兼容边界。

**复杂度：** 中低。适合当前项目，维护成本最低。

### 方案二：仅在 Web 页面增加正则提示

**核心思路：** Web 输入框使用 `/\b(BV[0-9A-Za-z]{10})\b/` 即时提取，服务端保持现有逻辑。

**优点：** 修改少、上线快。

**缺点：** 前端与服务端规则不一致；其他调用端无法复用；恶意域名中包含 BV 字符串时容易被误识别；无法解决服务端现有宽松校验。

**复杂度：** 低，但不符合共享来源约束，不推荐。

### 方案三：所有输入都实时提交服务端解析

**核心思路：** 对输入事件防抖后调用服务端，统一返回 BV 号。

**优点：** 前端逻辑最少，短链也能自动展开。

**缺点：** 每次输入可能产生请求；短链展开可能触发外网访问；响应乱序和限流处理更复杂；普通 BV/长链接没有必要走网络。

**复杂度：** 中高，收益不足，不推荐。

## 3. 推荐设计

采用方案一。

### 3.1 共享解析契约

协议层新增同步解析 API，返回以下三种结果之一：

- `resolved`：已得到严格合法的 BV 号和规范化 URL。
- `short-link`：输入是受支持的 `b23.tv` HTTP(S) 链接，需要服务端展开。
- `invalid`：空输入、非法 BV、非可信域名、无 BV 的链接或其他文本。

严格 BV 格式为 `BV` 加 10 位 ASCII 字母数字，总长度 12。输出统一保留 `BV` 前缀；后续 10 位保持原始大小写，因为 BV 编码大小写敏感。

仅在以下输入中提取 BV：

- 整个输入就是合法 BV。
- 主机名严格为 `www.bilibili.com`，路径为 `/video/<BV>`。
- 现有明确支持的 festival/watchlater URL，其 `bvid` 查询参数是合法 BV。

不从任意文本或非 B 站域名中搜索 BV 子串，避免把 `https://evil.example/?id=BV...` 当成可信链接。

### 3.2 Web 交互

- 对 `videoInput` 状态变化同步执行共享解析函数，不额外注册原生 `addEventListener`。
- 成功时在输入框下显示：`已识别视频ID：BV1Xs421N7Gr`。
- 输入为 `b23.tv` 时显示：`已识别 B 站短链接，点击“解析”获取视频信息`。
- 非空且无效时显示：`未能识别有效的B站视频链接`。
- 空输入不显示错误，避免页面初始即报错。
- 点击“解析”时仍发送原始输入给服务端；服务端是最终权威校验方。

### 3.3 服务端

- 复用协议层解析结果处理纯 BV 和长链接。
- `b23.tv` 仍通过现有服务端函数展开，限制主机名、协议和最大跳转次数。
- 展开后的 URL 必须再次走共享解析，不能直接信任重定向目标。
- 无效输入统一返回 `unsupported_bilibili_link`；Web 将其映射为“未能识别有效的B站视频链接”。
- 保留现有 AV 与番剧 `ep` 输入能力，避免本次需求意外破坏已支持功能；BV 即时提示只针对 BV，AV/番剧仍可点击解析。

## 4. 安全与性能

- 短链只允许 `http:`/`https:` 且主机严格等于 `b23.tv` 或 `www.b23.tv`。
- 每次用户操作只在点击“解析”时展开短链，最多 3 次跳转。
- 重定向结果仍需验证为支持的 B 站输入，防止开放重定向被用于访问任意业务 URL。
- 前端同步解析只做字符串和 URL 处理，不阻塞 UI。
- 不添加新数据库、WebSocket 消息或环境变量。

## 5. 测试与验收

### 协议层

- 接受纯 `BV1Xs421N7Gr`。
- 接受带追踪参数和尾部斜杠的标准链接，并移除无关参数。
- 拒绝不足/超过 10 位的 BV。
- 拒绝非字母数字字符。
- 拒绝非 B 站域名中包含 BV 的链接。
- 正确标识 `b23.tv` 短链。

### 服务端

- 短链跳转到合法视频链接后成功解析。
- 短链无 Location、跳转环、超过最大次数或跳到非支持地址时返回明确错误。
- 现有 AV、番剧和已授权视频解析行为不回归。

### Web

- 三个验收输入分别显示正确即时状态。
- `hello world` 显示指定错误文案。
- 输入为空不显示错误。
- 短链接输入时不自动发起 HTTP 请求，只有点击按钮才请求。

## 6. 变更范围

预计修改：

- `packages/protocol/src/video-ref.ts` 及测试：统一输入解析与严格 BV 校验。
- `server/src/web-routes.ts` 及 HTTP 测试：复用共享规则并验证短链边界。
- `web/src/App.tsx`、`web/src/styles.css` 及 Web 测试：即时识别提示和统一错误文案。

不修改扩展、不新增协议消息、不改变数据库或部署配置。

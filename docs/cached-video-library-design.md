# ECS 已缓存视频库与自动建房设计

## 1. 背景与目标

当前 B 站 CDN 播放地址会受到 IP、Referer、Cookie、签名和区域策略限制。浏览器直连可能返回 403，服务器代理又会受到香港 ECS 回源质量和公网出口带宽限制。

本方案改变媒体来源：管理员提前把可播放的视频文件放入 ECS 指定目录，网站自动展示这些文件。用户点击视频后，网页自动创建同步房间、开始播放，并生成可分享的房间链接。B 站 Cookie、CDN 签名和浏览器插件不再参与该播放链路。

目标链路如下：

```text
管理员下载/上传完成的视频
             |
             v
  /opt/bilisync/media/*.mp4
             |
       后台只扫描元数据
             |
             v
用户浏览器 <-- Nginx Range/206 -- ECS 本地文件
     |
     +------ WebSocket ------ Node 房间服务
```

### 1.1 用户故事

1. 管理员通过 SCP、rsync 或独立下载脚本把视频放入媒体目录。
2. 用户打开“已缓存视频”页面，看到当前可播放的视频列表。
3. 用户点击“播放并创建房间”。
4. 系统创建一个新房间，将所选视频设为共享视频并自动开始播放。
5. 页面展示邀请链接，其他用户打开链接后加入同一房间并同步播放。

### 1.2 本期范围

- 递归扫描指定目录中的 `.mp4` 文件。
- 提供缓存视频列表 API 和安全的文件播放入口。
- 增加“已缓存视频”页面及加载、空列表、不可用等状态。
- 点击视频后复用现有 `room:create`、`room:created` 和 `video:share` 流程自动建房。
- 复用现有网页播放器、WebSocket 同步和邀请链接机制。
- Nginx 支持 Range 请求和 `206 Partial Content`，保证长视频可拖动播放。

### 1.3 明确不做

- 不在网页中上传大文件。
- 不在应用内自动下载 B 站视频。
- 不做在线转码、转封装或转码任务队列。
- 不支持管理员在线编辑标题、标签和上下架状态。
- 不预先为每个视频创建永久房间；每次用户点击播放时创建新房间。
- 不在本期迁移到 OSS/CDN，但保留后续迁移空间。

## 2. 需求分析

### 2.1 涉及模块

| 模块                 | 变化                                                                   |
| -------------------- | ---------------------------------------------------------------------- |
| `server/src/config/` | 解析缓存视频根目录等环境变量                                           |
| `server/src/`        | 新增目录扫描和内存 catalog，提供列表及播放授权路由                     |
| `web/src/`           | 新增缓存视频列表视图及“播放并创建房间”编排                             |
| `deploy/nginx.conf`  | 新增仅内部可访问的本地媒体 location，由 Nginx 传输文件                 |
| `packages/protocol/` | 原则上不新增 WebSocket 消息；若提取共享 API 类型，只增加 HTTP 领域类型 |
| 测试与文档           | 增加扫描、安全、Range、建房流程和部署说明                              |

### 2.2 对现有功能的影响

- 现有 B 站解析、CDN 直连和代理回退功能保持不变，作为另一种视频来源继续可用。
- 现有房间协议版本不变，旧客户端仍可使用当前建房和加入房间流程。
- 缓存视频使用同源 URL，播放器不需要 CORS，也不需要 B 站 Referer 或 Cookie。
- 缓存视频被删除后，列表会在下次扫描时下线；已经打开的房间应显示“媒体已移除或不可用”，不能无限重试。

### 2.3 隐含要求与约束

- 下载过程中必须使用 `.part`、`.tmp` 等临时扩展名，下载完成后原子重命名为 `.mp4`，避免扫描到半成品。
- 视频必须优先使用浏览器兼容的 MP4/H.264/AAC，并把 `moov` 元数据移动到文件头，否则即使带宽足够也可能长时间黑屏。
- 应使用稳定的视频 ID，替换同一路径的视频文件时 ID 不变，现有分享状态不会因内容哈希变化而失效。
- API 和日志不得暴露 ECS 绝对路径。
- 目录遍历、符号链接逃逸和任意文件读取必须在服务端阻断。
- 多用户观看时，每位用户都单独消耗 ECS 出口带宽；本地缓存只解决 B 站 403 和回源不稳定，不会消除出口带宽瓶颈。

## 3. 方案对比

### 3.1 方案一：目录扫描 + 内存 Catalog + Nginx 本地文件发送（推荐）

#### 核心思路

Node 在启动时扫描媒体目录，并按固定周期增量刷新内存 catalog。浏览器通过 API 获取视频列表。播放请求先由 Node 按视频 ID 查找并校验文件，再返回 `X-Accel-Redirect`；实际文件读取、Range 处理和网络发送由 Nginx `sendfile` 完成。

```text
GET /api/web/cached-videos/:id/video.mp4
                   |
                   v
        Node 校验 id 和真实路径
                   |
        X-Accel-Redirect: /_cached-media/...
                   |
                   v
       Nginx internal + alias + sendfile
```

#### 技术原理

- catalog 只保存可公开的元数据和经 `realpath` 校验后的内部文件映射。
- `id` 基于规范化相对路径生成稳定摘要，不使用绝对路径或内容哈希。
- `/_cached-media/` 配置为 Nginx `internal`，浏览器不能绕过 Node 直接读取任意路径。
- Node 只处理短小的目录 API 和响应头，不在 JavaScript 进程中传输三小时视频。
- Nginx 原生处理 `Range`、`206`、`Content-Range`、`sendfile` 和客户端断开。

#### 优点

- 最少依赖，适合现有单台 ECS。
- 不依赖 B 站 Cookie、签名和 CDN 策略。
- 大文件传输不占用 Node 事件循环和应用内存。
- 可以复用现有直链 MP4 manifest、房间和同步协议。
- 后续切换 OSS/CDN 时，前端 catalog 和建房流程可以保持稳定。

#### 缺点

- catalog 默认不持久化，重启后需要重新扫描。
- 标题默认来自文件名，高级运营元数据能力有限。
- ECS 磁盘容量和公网出口带宽仍是硬限制。
- Nginx 部署配置必须与 Node 的媒体根目录保持一致。

#### 复杂度与维护性

- 实现复杂度：中。
- 对现有代码影响：中，集中在 server、web 和 Nginx 配置，不改变核心 WebSocket 协议。
- 维护性：高，数据源是文件系统，故障边界清晰。
- 当前项目适配度：最高，适合先交付可用 MVP。

### 3.2 方案二：持久化 Catalog（JSON/SQLite）+ Nginx 本地文件发送

#### 核心思路

媒体仍由 Nginx 发送，但扫描结果和人工元数据写入 JSON 或 SQLite。管理员可进一步设置标题、封面、排序、标签、隐藏状态和上下架时间。

#### 优点

- 元数据稳定，可支持人工运营和搜索筛选。
- 文件名变化、手工标题和上下架规则更容易管理。
- 视频数量较多时，查询和分页能力更好。

#### 缺点

- 需要处理数据库 schema、文件与记录一致性、删除和重命名冲突。
- 若本期没有管理后台，持久化数据缺少可靠的维护入口。
- 比用户当前目标多出明显的状态管理成本。

#### 复杂度与维护性

- 实现复杂度：中高。
- 对现有代码影响：中高，需要新增持久化层和一致性规则。
- 维护性：中，需要迁移、备份和异常恢复。
- 当前项目适配度：可选，适合后续需要视频运营后台时升级。

### 3.3 方案三：OSS/CDN 视频库 + ECS Catalog

#### 核心思路

视频文件存放到阿里云 OSS，并通过 CDN 或 OSS 域名播放。ECS 只维护 catalog、房间和 WebSocket，不承担视频出口流量。

#### 优点

- 并发能力和跨地域播放质量最好。
- ECS 出口带宽几乎不再受观众数量影响。
- 支持分片缓存、回源保护、生命周期和存储扩展。

#### 缺点

- 产生 OSS 存储、请求和 CDN 流量费用。
- 私有资源需要签名 URL，并处理有效期和刷新。
- 需要额外的同步或上传流程，部署和运维复杂度最高。

#### 复杂度与维护性

- 实现复杂度：高。
- 对现有代码影响：中，需要新增对象存储配置和签名服务，但前端变化不大。
- 维护性：高，云服务稳定，但需要成本和权限治理。
- 当前项目适配度：适合用户量上升后的生产方案，不建议作为第一版 MVP。

### 3.4 对比结论

| 维度          | 方案一：内存 Catalog + Nginx | 方案二：持久化 Catalog + Nginx | 方案三：OSS/CDN |
| ------------- | ---------------------------- | ------------------------------ | --------------- |
| B 站 403 依赖 | 无                           | 无                             | 无              |
| Node 转发视频 | 否                           | 否                             | 否              |
| ECS 出口带宽  | 高，随观众线性增长           | 高，随观众线性增长             | 低              |
| 元数据管理    | 基础                         | 完整                           | 完整            |
| 开发复杂度    | 中                           | 中高                           | 高              |
| 额外云成本    | 低                           | 低                             | 中高            |
| MVP 适合度    | 最佳                         | 一般                           | 一般            |

推荐先实施方案一。它以最小改动解决当前 B 站播放链路不可用的问题，并且不会把大文件流量压到 Node。若实际并发使 ECS 出口带宽成为瓶颈，再保持 catalog API 不变，把媒体存储迁移到方案三。

## 4. 推荐方案详细设计

### 4.1 配置

环境变量必须在 `server/src/config/` 集中解析：

| 环境变量                        | 默认值       | 说明                                             |
| ------------------------------- | ------------ | ------------------------------------------------ |
| `CACHED_VIDEO_DIR`              | 空，功能关闭 | ECS 上的媒体绝对目录，例如 `/opt/bilisync/media` |
| `CACHED_VIDEO_SCAN_INTERVAL_MS` | `30000`      | 目录重新扫描周期，设置合理上下限                 |

Nginx 的 `alias` 必须指向同一个物理目录。部署文档应明确这两个配置需要同步修改。功能关闭时列表 API 返回空列表或明确的 `feature_disabled`，不能影响现有服务启动。

第一版不增加更多可配置项；扩展名白名单固定为 `.mp4`，扫描周期只有确有部署需要时才开放配置。

### 4.2 Catalog 数据模型

建议在共享 HTTP 类型中定义：

```ts
type CachedVideoSummary = {
  id: string;
  title: string;
  streamUrl: string;
  posterUrl?: string;
  size: number;
  updatedAt: number;
  status: "ready";
};
```

说明：

- `id`：规范化相对路径的稳定摘要；不包含服务器绝对路径。
- `title`：默认取去掉扩展名的文件名。
- `streamUrl`：同源地址 `/api/web/cached-videos/:id/video.mp4`。
- `posterUrl`：可选，自动识别同名 `.jpg`、`.jpeg` 或 `.webp`。
- `size` 和 `updatedAt`：用于列表展示及浏览器缓存更新判断。
- 第一版不强制扫描时调用 `ffprobe`，避免新增生产依赖；视频时长由播放器加载 metadata 后获取。

内部 catalog 额外保存 `realPath`、规范化相对路径和文件 stat，但这些字段绝不返回浏览器。

### 4.3 扫描生命周期

1. 服务启动后执行首次全量扫描。
2. 只接收常规文件和 `.mp4` 扩展名，忽略隐藏文件、临时文件和目录。
3. 对每个候选执行 `realpath`，确认结果仍位于媒体根目录内。
4. 默认拒绝符号链接，防止链接到 `/etc` 或其他目录。
5. 用相对路径、大小和修改时间判断是否变化，原子替换 catalog 快照。
6. 每 30 秒重新扫描；单次失败记录不含绝对路径的错误，并保留上一份可用快照。
7. 服务关闭时清理定时器。

下载脚本应遵循：

```bash
# 下载阶段不进入 catalog
download-tool ... -o movie.mp4.part

# 下载完成后先转为浏览器友好的 faststart 文件，再原子上线
ffmpeg -i movie.mp4.part -c copy -movflags +faststart movie.ready.mp4
mv movie.ready.mp4 movie.mp4
```

若源编码不是 H.264/AAC，需要离线转码，而不是只使用 `-c copy`。

### 4.4 HTTP 接口

#### 获取列表

```http
GET /api/web/cached-videos
Cache-Control: no-store
```

响应：

```json
{
  "ok": true,
  "data": {
    "videos": [
      {
        "id": "cv_...",
        "title": "示例视频",
        "streamUrl": "/api/web/cached-videos/cv_.../video.mp4",
        "size": 2700000000,
        "updatedAt": 1784000000000,
        "status": "ready"
      }
    ]
  }
}
```

#### 播放文件

```http
GET /api/web/cached-videos/:id/video.mp4
Range: bytes=...
```

Node 根据 `id` 从当前 catalog 查找文件，校验仍然存在后设置：

```http
X-Accel-Redirect: /_cached-media/<安全编码的相对路径>
Content-Type: video/mp4
```

随后由 Nginx 返回 `200` 或 `206`。不存在、已删除或不在 catalog 的 ID 返回 `404`。

### 4.5 Nginx 媒体配置

设计目标是禁止外部直接访问内部路径，并让 Nginx 原生发送文件：

```nginx
location ^~ /_cached-media/ {
    internal;
    alias /opt/bilisync/media/;
    sendfile on;
    aio threads;
    tcp_nopush on;
    add_header Accept-Ranges bytes always;
    add_header Cache-Control "private, max-age=3600" always;
}
```

最终配置需在目标 Linux/Nginx 版本验证 `aio threads` 是否可用；若模块未编译，则删除该行，`sendfile` 仍可工作。不得启用 `autoindex`。

### 4.6 点击视频后的自动建房流程

```text
用户点击“播放并创建房间”
        |
        v
Web 保存 pendingCachedVideoId
        |
        v
发送现有 room:create
        |
        v
收到现有 room:created（得到 roomCode/memberToken/joinToken）
        |
        v
用同源 streamUrl 调用现有 /api/web/video/resolve
        |
        v
发送现有 video:share，currentTime=0，playState=playing
        |
        v
播放器加载 + 页面展示现有邀请链接
```

Web 端应把现有“解析并分享视频”的逻辑提取为可接收显式 session 和 URL 的小型 helper，供手工分享和缓存视频自动建房共同复用。不要复制一套 `video:share` 组包逻辑。

边界规则：

- 用户连续点击时，只允许一个建房请求处于进行中，按钮显示加载状态。
- 建房失败时清理 pending 选择，保留在视频列表并显示可重试错误。
- 视频在建房后被删除时，播放接口返回 404，播放器显示“该缓存视频已移除”。
- 每次点击创建新房间；本期不查找或复用其他用户已创建的房间。
- 自动播放仍受浏览器策略限制；若浏览器阻止有声自动播放，显示标准播放按钮，不把它当作建房失败。

### 4.7 “已缓存视频”页面

页面作为实际可用的视频库视图，不做营销落地页：

- 顶部保留当前产品导航和房间入口。
- 主区域使用紧凑的视频列表或网格，显示封面、标题、文件大小和更新时间。
- 每项提供清晰的“播放并创建房间”命令。
- 空列表显示“暂无已缓存视频”，并保留现有手工输入视频入口。
- 加载失败、功能未配置、视频已删除分别显示明确错误，不混成“网络错误”。
- 移动端保证标题换行、按钮不溢出，封面使用固定宽高比避免布局跳动。

## 5. 性能与成本评估

### 5.1 带宽

三小时视频按平均 `2 Mbps` 估算：

```text
2 Mbps / 8 × 10,800 秒 ≈ 2.7 GB/用户
```

并发出口需求约为：

| 同时观看人数 | 2 Mbps 视频的最低持续出口 | 建议预留 30% 后 |
| -----------: | ------------------------: | --------------: |
|            1 |                    2 Mbps |        2.6 Mbps |
|            3 |                    6 Mbps |        7.8 Mbps |
|            5 |                   10 Mbps |         13 Mbps |
|           10 |                   20 Mbps |         26 Mbps |

如果 ECS 只有 5 Mbps 固定带宽，三个用户同时观看 2 Mbps 视频就会持续卡顿。Nginx、Range、缓存和代码优化无法突破公网出口上限。上线前必须确认 ECS 实际带宽计费模式、峰值和跨境用户网络质量。

### 5.2 CPU、内存与磁盘

- Node 扫描只读取目录项和 stat，正常规模下 CPU/内存开销很低。
- Nginx `sendfile` 避免用户态复制，CPU 开销显著低于 Node 流式转发。
- 每个并发连接仍会占用 Nginx 连接和内核页缓存，但不会把完整视频读入内存。
- 磁盘必须预留下载临时文件、faststart 输出文件和最终文件同时存在的空间；处理单个 3 GB 视频时可能临时需要约 6 至 9 GB。

### 5.3 扩容触发条件

出现以下任一情况，应迁移到 OSS/CDN，而不是继续优化 Node：

- 峰值视频出口长期超过 ECS 带宽的 70%。
- 两个以上地区用户持续出现下载速度低于视频码率。
- 视频库容量接近 ECS 数据盘的 70%。
- 单机故障或扩容期间需要保持媒体可用。

## 6. 安全设计

- 媒体根目录必须是显式配置，不能默认扫描项目目录或系统目录。
- catalog ID 只能查表，禁止把请求参数直接拼到文件路径。
- 所有候选文件经过 `realpath` 根目录包含校验。
- 拒绝符号链接、非普通文件、未知扩展名和临时文件。
- `/_cached-media/` 必须使用 `internal`，禁止目录列表。
- API、访问日志和错误信息不记录 ECS 绝对路径。
- 对文件名进行安全编码后再写入 `X-Accel-Redirect`，拒绝换行、空字节和非法路径片段。
- 默认把缓存视频视为站点成员可访问的内容。如果未来需要私有访问，应给播放 API 增加房间成员令牌校验或短期签名，不能仅依赖不可猜测 ID。
- 管理员必须确认视频内容具有合法的存储和分发授权。

## 7. 错误处理

| 场景               | 行为                                                |
| ------------------ | --------------------------------------------------- |
| 未配置媒体目录     | 列表显示功能未配置，现有房间功能不受影响            |
| 目录不存在或不可读 | 服务记录受控错误，列表 API 返回明确错误             |
| 单个文件不可读     | 忽略该文件，其他视频继续展示                        |
| 扫描中途失败       | 保留上一份成功快照，下轮重试                        |
| 文件下载未完成     | `.part`/`.tmp` 不进入 catalog                       |
| 文件在播放前被删除 | 播放接口返回 404，前端提示已移除                    |
| 文件编码不兼容     | 播放器提示格式不受支持；管理员离线转码              |
| 浏览器阻止自动播放 | 房间创建成功，提示用户点击播放                      |
| ECS 带宽不足       | 记录 stalled/超时现象并提示；运维扩容或迁移 OSS/CDN |

## 8. 预计代码改动

实施方案一时，预计新增或修改以下位置，最终以现有模块边界为准：

- `server/src/config/cached-video-config.ts`：集中解析和校验环境变量。
- `server/src/cached-videos/`：目录扫描器、catalog 和路径安全校验。
- `server/src/web-routes.ts` 或独立路由模块：列表和 `X-Accel-Redirect` 响应。
- `server/src/index.ts`：只做扫描器依赖装配和生命周期接线，保持入口轻量。
- `packages/protocol/src/`：缓存视频 HTTP 数据类型及导出（若当前 Web/Server 类型共享方式适用）。
- `web/src/`：缓存视频 API client、列表视图和自动建房状态编排。
- `deploy/nginx.conf`：内部媒体 location。
- `docs/reference/security-env*.md`：环境变量和安全说明。
- 对应 server、web 和 protocol 测试文件。

不修改扩展插件同步逻辑，不删除现有 B 站播放能力，不迁移数据库。

## 9. 测试与验收

### 9.1 服务端测试

- 扫描只返回 `.mp4` 常规文件。
- `.part`、目录、符号链接和根目录外文件不会进入 catalog。
- ID 稳定，替换同路径文件后 ID 不变，元数据更新。
- API 不返回绝对路径。
- 未知或已删除 ID 返回 404。
- 合法播放请求返回正确的 `X-Accel-Redirect` 和 MIME 类型。
- 扫描失败保留上一份成功快照。
- 功能未配置时不影响服务启动。

### 9.2 Web 测试

- 正确展示加载、空列表、成功和错误状态。
- 点击视频只发送一次 `room:create`。
- 收到 `room:created` 后解析同源 MP4 并发送一次 `video:share`。
- 建房或解析失败会清理 pending 状态并允许重试。
- 建房成功后显示邀请链接并开始加载所选视频。
- 连续点击不会创建多个意外房间。
- 浏览器拒绝自动播放时仍保留已创建房间。

### 9.3 部署验收

- 将 `.mp4.part` 放入目录时网页不展示；重命名为 `.mp4` 后在一个扫描周期内出现。
- 浏览器 Network 面板中视频请求指向同源缓存视频 API，不访问 B 站 CDN。
- 初始播放返回 `200` 或 `206`，拖动到三小时视频后半段返回有效 `206 Content-Range`。
- 两名用户通过邀请链接进入同一房间，播放、暂停、拖动同步正常。
- Node 进程内存不会随已发送视频字节持续增长。
- Nginx 日志显示媒体字节由内部 location 发送。
- 在目标并发下，ECS 出口带宽高于视频总码率并保留至少 30% 余量。

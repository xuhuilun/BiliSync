# Web 用户头像兜底设计

## 需求分析

目标是在 Web 页面无法取得 B 站用户头像时，使用用户提供的粉色 Bilibili 图片作为默认头像。

“无法获取”包括：

- 登录状态接口未返回 `avatarUrl`。
- 返回了头像地址，但浏览器加载失败、超时或资源失效。
- 用户未登录，当前头像按钮也使用同一张默认图片，保持视觉一致。

本次只修改 Web 静态资源和头像渲染，不改变登录接口、服务端数据或协议。

## 方案一：本地静态资源 + `onError` 回退（推荐）

### 核心思路

将用户提供的 PNG 复制到 `web/public/`，头像始终渲染 `<img>`：有远程地址时优先加载远程头像，没有地址时直接加载本地图片；远程图片触发 `onError` 后切换到本地图片。

### 优点

- 同时覆盖“字段缺失”和“图片加载失败”。
- 默认图片随 Web 构建部署，不依赖外部网络。
- 改动范围小，行为直观，容易测试和维护。

### 缺点

- 构建产物增加一张约 128×128 的 PNG，体积影响可忽略。
- 需要防止兜底图片自身失败时反复触发 `onError`。

### 复杂度与影响

复杂度低，只影响 `web/public/`、头像渲染和对应测试，适合当前项目。

## 方案二：CSS 背景图兜底

### 核心思路

把默认图片设为头像容器背景，有远程头像时在上层渲染 `<img>`。

### 优点

- 远程图片透明或未完成加载时仍能看到背景。

### 缺点

- `<img>` 加载失败后仍可能显示破图图标。
- 无障碍替代文本和加载状态更难表达。
- CSS 与渲染逻辑耦合，不如显式回退清晰。

### 复杂度与影响

复杂度低，但维护性弱于方案一，不推荐。

## 推荐实现

采用方案一：

1. 将源图片复制为 `web/public/default-avatar.png`，不修改原始临时文件。
2. 新增一个小型头像 URL/错误回退辅助逻辑，避免继续扩大 `App.tsx` 的条件分支。
3. 无 `avatarUrl` 时渲染 `/default-avatar.png`。
4. 远程头像加载失败时将当前图片源替换为 `/default-avatar.png`；若当前已经是默认头像，则不重复替换。
5. 保留有意义的 `alt` 文本，例如“用户头像”。
6. 测试覆盖缺少 URL、远程 URL 正常使用和加载失败回退。

## 预计修改文件

- 新增 `web/public/default-avatar.png`。
- 新增或修改 Web 头像辅助模块及测试。
- 修改 `web/src/App.tsx` 的头像渲染。
- 必要时微调 `web/src/styles.css`，确保默认图使用 `object-fit: cover`。

## 验收标准

- 未登录或接口没有头像 URL 时显示指定图片。
- 有有效远程头像时优先显示远程头像。
- 远程头像加载失败后自动切换指定图片，无无限错误循环。
- 登录、二维码弹窗、房间和视频播放功能不受影响。

# 同房间实时语音聊天：技术与产品设计方案

## 1. 需求分析与边界

### 1.1 核心目标

在现有 Bili-SyncPlay 房间内增加低延迟实时语音，使成员在观看 B 站视频时可以开麦交流，并让语音成员身份、权限和房间生命周期与现有房间保持一致。

### 1.2 首期假设

- 首期以桌面 Chrome/Edge 的普通 HTTPS Web 页面为主，页面内使用 `<video>` 元素播放视频；Firefox 后续验证，不承诺移动端和 Safari。
- 每房典型 2～8 人，建议硬上限 12 人；只传音频，不传摄像头、屏幕共享或录音。
- 目标端到端语音延迟：正常网络下 P95 小于 300 ms。
- RTC 厂商负责音频媒体传输；现有 WebSocket 服务仍是房间与权限的权威状态源。
- 首期不做服务端录音、内容审核和语音转写。若公开运营，需另行补充隐私协议、举报和风控设计。

### 1.3 可验证验收指标

- 2～8 人房间内，加入语音成功率不低于 99.5%，正常网络 P95 延迟小于 300 ms。
- 用户能授权麦克风、加入/退出语音、静音/解除静音、按键说话、独立调节成员音量。
- 房主可全员禁言、禁止指定成员发言和恢复权限；服务端校验权限，不能只依赖前端 UI。
- WebSocket 短暂重连不导致重复入会；最后一名成员离房后，频道不再产生可计费订阅。
- RTC 故障不影响原有视频同步、文字状态和房间生命周期。

## 2. 产品功能设计

### 2.1 建议的语音模式

| 模式            | 行为                                             | 首期建议                                                                 |
| --------------- | ------------------------------------------------ | ------------------------------------------------------------------------ |
| 自由麦          | 用户手动开/关麦；打开后由 VAD 判断是否正在说话   | 必做，默认模式                                                           |
| 按键说话（PTT） | 按住快捷键发送音频，松开静音                     | 必做；仅页面聚焦且焦点不在输入框、编辑器等可输入控件时响应，快捷键可配置 |
| 房主控麦        | 房主可禁止指定成员发言、恢复发言                 | 必做                                                                     |
| 全员禁言        | 房主一键禁止普通成员上行音频；可选择仅房主可说话 | 必做                                                                     |
| 申请发言/排麦   | 成员申请、房主批准，形成麦序                     | 二期；小房间首期收益有限                                                 |
| 自动语音激活    | 根据音量门限自动开关上行轨道                     | 不建议首期；易截断句首、受环境噪声影响                                   |

“静音”必须区分三个状态：用户主动闭麦、房主禁止发言、麦克风权限/设备异常。房主禁言后，客户端应立即停止发布音轨；服务端权限状态同时阻止重新取得有效 RTC 凭证或重新发布。

### 2.2 UI 状态表达

- 成员头像外圈：灰色表示未加入语音，绿色表示已入会且可发言，红色斜杠表示静音，锁形图标表示被房主禁言。
- 说话态：由本地/远端音频音量回调驱动头像亮圈或 3 级声波，使用约 150～300 ms 平滑和衰减，避免动画抖动。
- 自己的状态始终放在语音面板顶部，并显示当前输入设备、实时电平和麦克风权限错误。
- 网络质量使用小型信号图标，只在明显恶化时提示“网络不稳定”，避免长期展示复杂 RTC 指标。
- 无障碍：颜色之外同时使用图标和文本；动效遵循 `prefers-reduced-motion`。

### 2.3 单成员音量

建议支持。每个远端成员提供 0～200% 音量滑块和“仅对我静音”。设置保存在浏览器 `localStorage` 或现有前端持久化层，以稳定的用户 ID 为键；成员退出、重进后继续生效。默认 100%，超过 100% 时提示可能失真。该操作仅改变本地播放增益，不应写入房间共享状态。

## 3. 核心技术方案对比

### 3.1 方案一：第三方 RTC 厂商（推荐）

**核心思路：** 使用声网、ZEGO 或腾讯云 RTC 的音频房间能力。现有服务器签发短期 RTC Token、维护权限和房间映射；厂商媒体节点负责接入、转发、弱网对抗和 TURN。

**技术原理：** 客户端加入与 Bili-SyncPlay 房间一一映射的 RTC 频道，发布单声道 Opus 麦克风轨道，订阅其他成员音轨。WebSocket 只承载业务状态和控制指令，不转发音频。

**优点：** 单台 ECS 几乎不承担媒体带宽；全球/全国节点、弱网优化、监控与 SLA 成熟；上线快；小规模按量付费通常比维护 SFU 更划算；延迟通常可控制在 100～300 ms。

**缺点：** 持续产生分钟数费用；有厂商锁定；Token、回调和合规配置增加集成工作；不同厂商的 Web SDK、浏览器兼容性、包体和部署 CSP 要求需做原型验证。

**复杂度：** 中。对协议、服务端鉴权、Web 页面状态管理和语音 UI 都有增量修改，但无需运维媒体集群。

**维护性与适配：** 最适合当前“单 ECS + 企业级稳定”的约束。建议先对声网、ZEGO、腾讯云各做一个最小 Web 页面 PoC，以实际报价、目标用户地域、SDK 包体/CSP、音量回调及控制能力选厂商，不在设计阶段凭品牌武断定标。

### 3.2 方案二：WebRTC P2P 网状架构

**核心思路：** 每个成员与房内其他成员建立一条 PeerConnection，现有 WebSocket 转发 SDP/ICE 信令，必要时自建或购买 TURN。

**优点：** 2～4 人时架构直观；媒体不经过应用 ECS（TURN 中继除外）；无需支付 RTC 音频分钟费；数据路径直接时延较低。

**缺点：** 每个用户上行带宽和编码负担随人数按 `N-1` 增长；连接数按 `N×(N-1)/2` 增长；企业网络/NAT 下必须依赖 TURN；网络质量由最差端拖累，监控、重连和兼容性成本高。8～12 人已不适合网状架构。

**复杂度：** 表面中等、生产化偏高。需要 ICE/TURN、协商冲突处理、重连、设备切换、浏览器差异和可观测性。

**影响与适配：** 会显著增加 WebSocket 信令逻辑；不适合“企业级稳定”和可增长人数目标。只适合明确限制为 4 人以内、预算极低的私人部署。

### 3.3 方案三：自建 SFU（MediaSoup）

**核心思路：** 在自有基础设施运行 MediaSoup，客户端只向 SFU 发布一条音频，SFU 将各轨道转发给订阅者；搭配 coturn、监控和区域部署。

**优点：** 控制力强、可定制路由/录音/审核；规模起来后可降低厂商分钟费；避免深度绑定厂商。

**缺点：** 单 ECS 是单点故障；公网出口带宽成为核心成本和瓶颈；需要部署 coturn、证书、端口、防火墙、容量控制、滚动升级、故障迁移和媒体质量监控。SFU 虽不必解码混音，但仍承担大量包转发和加密开销。单机无法自然满足企业级稳定。

**复杂度：** 高。需要独立媒体服务、worker 管理、信令适配、容量调度和至少双节点高可用。

**影响与适配：** 长期大规模、团队具备 RTC 运维能力时合适；当前阶段不推荐。可在语音月费用稳定高于自建多节点总拥有成本后重新评估。

### 3.4 结论

| 维度           | 第三方 RTC | P2P Mesh           | 自建 SFU              |
| -------------- | ---------- | ------------------ | --------------------- |
| 2～8 人稳定性  | 高         | 中到低             | 中（单机）/高（集群） |
| P95 < 300 ms   | 易达成     | 小房间可达成       | 合理部署可达成        |
| ECS 媒体带宽   | 接近 0     | 接近 0，TURN 例外  | 高                    |
| 初期研发/运维  | 中/低      | 中/中高            | 高/高                 |
| 初期成本       | 按量、可控 | 低，但 TURN 有成本 | 机器、带宽和人力高    |
| 当前项目适合度 | **最高**   | 仅超小房间         | 当前不适合            |

最终推荐方案一：第三方 RTC。它把最难稳定的媒体链路交给专业网络，现有 ECS 只负责轻量鉴权和控制，最符合现阶段的风险、成本和延迟目标。

## 4. 音频混合、播放与闪避

### 4.1 混音策略

浏览器本身会把页面内 `<video>` 原声和 RTC 远端音频共同送往系统输出，基础场景无需服务端混音。为实现单成员音量和闪避，建议使用 Web Audio API：每个远端音轨连接独立 `GainNode`，再连接统一语音总线；`<video>` 通过 `MediaElementAudioSourceNode` 接入独立视频总线。两条总线最终连接到同一个 `AudioContext.destination`。

页面必须使用 HTTPS（`localhost` 开发环境除外），并在用户点击“加入语音”等明确手势后申请麦克风权限和调用 `AudioContext.resume()`，以满足浏览器权限与自动播放策略。同一个 `<video>` 元素只能调用一次 `createMediaElementSource()`，因此媒体源节点应由页面级音频控制器统一创建和复用。

如果 `<video>` 加载的是跨域媒体，媒体服务器必须返回正确的 CORS 响应，页面也需按资源要求设置 `crossorigin`；否则接入 Web Audio 后可能输出静音。实施前必须用真实视频源做 PoC。若无法可靠接管原声，可退化为调节 `video.volume`，但要保存用户原始音量，并防止与播放器自身音量操作互相覆盖。

### 4.2 音频闪避建议

建议提供，默认开启且允许关闭，而不是强制。远端任一成员进入“正在说话”状态后，将视频音量平滑降低约 6～10 dB；无人说话持续 500～800 ms 后，用 300～600 ms 恢复。攻击时间约 80～150 ms，避免突然抽动。

前端可用 RTC SDK 的远端音量指示或 `AnalyserNode` 做 VAD：

1. 对每个远端轨道计算平滑后的 RMS/音量等级。
2. 连续超过门限约 100～200 ms 才判定说话。
3. 多人同时说话只触发一次视频总线衰减，不重复叠加。
4. 通过 `GainNode.gain.setTargetAtTime` 或 `linearRampToValueAtTime` 平滑调整。
5. 保存用户原视频音量，仅施加临时乘数；退出语音、页面切换或异常时必须恢复。

本地麦克风采集建议启用浏览器/SDK 的 AEC、NS、AGC（回声消除、降噪、自动增益）。推荐耳机；扬声器外放时，AEC 是避免 `<video>` 原声被重新采进麦克风的关键。

## 5. 与现有架构融合

### 5.1 职责边界

- `@bili-syncplay/protocol`：新增语音状态、权限与控制消息类型/守卫；不放厂商 SDK 类型。
- WebSocket 服务：仍是房间成员、房主身份、禁言权限和 RTC 凭证签发的权威源。
- RTC 厂商：仅负责实时音频传输与媒体质量，不替代业务房间。
- Web 页面：建议新增独立语音控制器或服务层，管理 Token、RTC 会话、设备轨道、重连、权限及 Web Audio 节点；UI 组件不直接持有厂商 SDK 客户端。
- 页面 UI：分离语音状态、用户动作和渲染，避免把 RTC SDK 调用直接塞进 DOM 事件处理。组件卸载不等同于业务离房时，不应误关全局语音会话。

### 5.2 推荐交互流程

1. 用户先通过现有 WebSocket 加入业务房间。
2. 用户点击“加入语音”，客户端向服务端请求 RTC 凭证。
3. 服务端校验房间成员身份和禁言状态，签发短期 Token；频道 ID 使用不可猜测的内部 room ID 映射，RTC UID 使用稳定且不泄露隐私的映射值。
4. 客户端加入 RTC 频道。默认先订阅远端音频；只有用户明确授权并开麦后才采集和发布麦克风。
5. 静音、说话态和网络质量通过 RTC SDK 获取；对其他业务成员展示所需的稳定状态可经 WebSocket 广播，但高频音量值不要走现有 WebSocket。
6. 房主禁言时，WebSocket 广播权威权限变更；目标客户端停止发布。服务端还需通过厂商服务端能力撤销发布权限或刷新受限 Token，防止修改客户端绕过。
7. 离开业务房间时先退出 RTC；RTC 失败不阻塞 WebSocket 离房。

页面刷新、关闭标签页和浏览器崩溃时，不能只依赖 `beforeunload` 或 `sendBeacon` 完成退会；客户端应尽力清理，但服务端必须通过 WebSocket 心跳、断线宽限和 RTC 厂商事件做最终回收。

### 5.3 生命周期与资源释放

不要仅用“业务房间人数”判断 RTC 资源，应单独维护 `voiceParticipantCount` 或语音成员集合：

- 用户退出语音：取消发布、退 RTC 频道、清理本地音频节点和设备轨道。
- 用户离开业务房间/被踢：服务端撤销其 RTC 权限，客户端执行同样清理。
- WebSocket 断开：设置约 30～60 秒重连宽限，避免网络抖动导致频繁退会；超过宽限则服务端将其标记离房并撤销 RTC 权限。
- 最后一名语音成员离开：第三方 RTC 通常无需显式销毁频道，无发布/订阅即停止计费；服务端删除临时映射、计数和权限状态。
- 最后一名业务成员离开：沿用现有房间销毁流程，并幂等清理所有 RTC 状态。

必须设计幂等的加入/离开操作和 session generation，防止旧连接的迟到事件清掉新会话。厂商 webhook 只作为核对与回收依据，不应成为实时 UI 的唯一来源。

### 5.4 故障隔离

- RTC Token 接口限流并记录审计信息；密钥只放服务端配置，不进入 Web 前端构建产物。
- RTC 服务不可用时展示“语音暂不可用”，原视频同步继续工作。
- 厂商 SDK 懒加载或独立打包，避免拖累未加入语音的页面用户启动。
- 建议观测：入会成功率、入会耗时、断线率、P95 RTT/丢包、活跃音频分钟、房间人数和 Token 签发失败。

## 6. 资源与成本评估

### 6.1 编码假设

推荐 Opus 单声道、20 ms 包间隔：语音码率 24～32 kbps；考虑 RTP/UDP/SRTP/IP 头部和重传，按每路 35～55 kbps 的实际网络流量估算。若强调音乐质感可提高到 48 kbps；普通聊天没有必要使用 48 kHz 立体声高码率。

### 6.2 第三方 RTC 对现有 ECS 的影响

- **带宽：** ECS 不承载音频，只增加 Token 请求、WebSocket 控制消息和少量状态，通常远低于 1 kbps/活跃用户均值，可忽略于媒体流量。
- **CPU/内存：** 主要是签名计算、状态维护和监控，通常是现有服务的轻微增量。准确值应通过压测得到，不应预先承诺固定百分比。
- **终端流量：** 每个开麦用户上行约 35～55 kbps；每个用户下行约 `(N-1) × 35～55 kbps`。8 人房间单用户下行约 245～385 kbps。
- **费用：** 多数厂商按订阅/音频分钟或套餐计费，地区和计费规则差异很大。预算公式可用：`月活跃语音人分钟 × 音频单价 + 增值服务 + 超额流量`，签约前以厂商当期报价和“发布/订阅是否重复计费”规则核算。

### 6.3 若自建 SFU 的量级估算

对 N 人全部开麦且全部订阅，SFU 媒体出口近似 `N × (N-1) × 每路码率`，入口近似 `N × 每路码率`。按每路 45 kbps：

| 房间人数 |  SFU 入口 |  SFU 出口 |  双向合计 |
| -------- | --------: | --------: | --------: |
| 4        | 0.18 Mbps | 0.54 Mbps | 0.72 Mbps |
| 8        | 0.36 Mbps | 2.52 Mbps | 2.88 Mbps |
| 12       | 0.54 Mbps | 5.94 Mbps | 6.48 Mbps |

这只是持续媒体有效流量的粗估，未包含 TURN、重传、监控和峰值冗余。100 个并发 8 人满房约需 252 Mbps SFU 出口，因此单 ECS 公网带宽很快成为瓶颈。MediaSoup 音频转发的 CPU 通常比服务端混音低，但 DTLS/SRTP、包转发和连接管理仍需实测；生产环境还需至少双节点、独立 TURN 和容量调度，不能用单一 CPU/内存数字代替压测。

### 6.4 节流策略

- 只传单声道 Opus 音频，默认 24～32 kbps；不传视频。
- 启用 DTX/VAD，让静音或无语音时减少包量；但保留舒适噪声并验证不同 SDK 行为。
- 静音时真正停止/禁用上行轨道，而不是只把本地音量设为零。
- 只订阅当前业务房间成员，离房立即取消订阅。
- 设每房人数上限；大房间采用房主控麦/少数发言者模式，避免所有人持续发布。
- 不建议为省流量把采样率强行降到 8/16 kHz。Opus 24～32 kbps 单声道已足够经济，过低采样率会明显损害体验；让 SDK 做自适应码率更稳妥。

## 7. 最终落地建议

### 7.1 推荐方案

采用“**现有 WebSocket 作为权威控制面 + 第三方 RTC 作为媒体面**”：

- MVP 支持自由麦、PTT、单人音量、房主单人禁言和全员禁言。
- 音频使用 Opus 单声道 24～32 kbps，启用 AEC/NS/AGC 与 DTX。
- 视频闪避默认开启，可关闭；默认衰减 8 dB，并使用平滑攻击/释放。
- 每房建议上限 12 人；典型体验针对 2～8 人优化。
- 服务端签发短期、房间绑定、用户绑定的 RTC Token，密钥不下发。
- 先做厂商 PoC 与一轮 2/4/8/12 人弱网测试，再根据实测和商务报价在声网、ZEGO、腾讯云中定标。

### 7.2 分期路线

**阶段 A：技术 PoC**

- 验证 Web SDK 与现有前端构建/CSP 的兼容性、麦克风授权、浏览器自动播放策略、设备切换、`<video>` 跨域音频接入、单成员增益、AEC 和 P95 延迟。
- 用相同测试脚本比较三家候选厂商，不接入正式房间协议。

**阶段 B：MVP**

- 接入 Token、语音房间生命周期、自由麦/PTT、音量控制、房主禁言、故障隔离和基础指标。
- RTC 失败时保证现有同步功能无回归。

**阶段 C：生产增强**

- 加入 webhook 对账、滥用控制、质量看板、费用告警、设备选择以及 Firefox/Safari 兼容性验证。
- 根据真实使用数据决定是否需要排麦、语音激活或迁移/多厂商容灾。

### 7.3 备选方案的适用条件

- 若产品永久限制在 4 人以内、仅私人部署且可接受 TURN/连接稳定性风险，可选 P2P Mesh。
- 当月 RTC 成本长期显著超过“至少双 SFU + TURN + 带宽 + 监控 + 运维人力”的总拥有成本，并且团队已有 RTC 运维能力时，再考虑 MediaSoup。
